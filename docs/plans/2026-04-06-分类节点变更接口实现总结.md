# 文件分类节点变更接口实现总结

本方案旨在根据 `docs/接口文档/分类节点变更.md` 的要求以及补充需求（严格一致性校验及工作区联动迁移），在服务端实现 `/llm/reassign` 接口以支持文档分类节点（`architectureId`）的无缝迁移与变更。

## 1. 核心实现内容

本项目重点完成的内容包括以下三大核心模块：

### 1.1 核心路由接口 `POST /llm/reassign`
- **代码位置**：[llm.py](file:///c:/Files/Codes/DocSense/app/blueprints/llm.py)
- **校验逻辑**：解析并校验入参 JSON Payload 不得为空。请求接收后会调用 `kb_service.get_document_record` 获取数据库原始记录。若查无记录，返回业务级失败响应。
- **严格一致性阻断**：在执行更新前验证现有持久化记录的分类是否与当前请求宣称的 `oldArchitectureId` 完全匹配。若原有节点的 ID 与请求不一致，直接预先拦截并返回对应错误消息反馈，以实现数据安全的强一致性要求。

### 1.2 AnythingLLM 知识库空间联动
由于不同的体系架构树划分了不同的 Workspace 工作区域，该接口在转移系统节点属性时，同步引入了操作 AnythingLLM 底层 RAG 管理域的联动行为：
- **无工作区保底创建**：依据 `kb_service.get_workspace_slug` 拿取新体系工作区标识。若请求对应的新架构域尚未注册创建该体系的工作区时，调用 `client.create_rag_workspace` 动态构建 RAG 限定的新工作区实例。并在本地底层的 `workspaces` 落表。
- **关联向量更新（移出旧库+嵌入新库）**：获取该文件先前的旧工作区标识以及系统内的 `doc_path`；直接调用批处理借口 `client.update_embeddings_batch(deletes=...)` 将其解除检索特征。随后发起 `client.update_embeddings(...)` 请求将其全新挂载嵌入到新的工作区域建立可供索引的新标识引用。

> [!TIP]
> **(Hotfix重点) 路径清洗Bug修复**：在验证解绑操作时发现旧域资源未消失的问题。经排查是因为早期向 `update_embeddings_batch` 传的 `doc_path` 携带有 `C:\...\custom-documents\xxx` 格式的本地前导绝对物理路径，而 AnythingLLM 服务端执行删库指引时强制且仅认类似于内部形式的精确相对路径（`custom-documents/xxx`）。目前已经在 `anythingllm_client.py` 底座抽离和引入了公共的 `_clean_doc_path` 工具流处理。确保所有的解绑执行完全精确转化，经系统验证该 Bug 已闭环修复脱模率达到100%。

### 1.3 数据层映射更新
- **代码位置**：[database.py](file:///c:/Files/Codes/DocSense/app/services/core/database.py)
- **逻辑增加**：在 `DatabaseService` 类追加了 `update_document_architecture(self, file_name, new_architecture_id)` 方法，通过事务级操作对应的修改 `documents` 映射底表中的结构体系绑定归属指令；此类变通更新已部署在该类一贯具备的安全锁管理区间内执行保障了线程安全。

## 2. 测试与验证

在针对路由的主测文件 [`test_routes.py`](file:///c:/Files/Codes/DocSense/tests/test_routes.py) 模块中增补并彻底覆盖了新业务规则场景所需的完整分支：

- **容错校验**：涵盖未传递/非法传递的分支校验及找不到文件报错的基座控制。
- **一致性验证阻断**：追加并运行了 `test_reassign_returns_error_when_inconsistent` 等测例，实证 Old 节点不符情况下的非法越界阻断性能。
