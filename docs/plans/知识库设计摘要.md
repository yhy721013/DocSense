# 知识库存储与架构设计说明

本文档旨在对当前业务系统中检索增强生成（RAG）功能底层的存储机制与设计决策进行总结。当前系统基于 AnythingLLM 构建大模型知识分类能力，并借助本地 SQLite 进行业务控制层的双轨管理隔离。

---

## 一、 AnythingLLM 核心机制与原理

### 1. 文件的提取与向量化机制 (RAG)
AnythingLLM 的核心机制是检索增强生成（RAG）。当文件（如装备资料）上传时系统会执行严格的转化：
- 系统解析文件并将其切割成文本片段（Chunk），通过嵌入模型（Embedding Model）转化为向量数字数组，存储在底层的向量数据库（如 LanceDB、Chroma 等）中。
- 当查询时，系统同样会将提问转化为向量进行相似度检索，从而精准提取最相关的 Chunk 作为上下文喂给大模型。
- **Chunk的生成变量**：相同的文件进入向量数据库生成的 Chunk 极大概率是不同的。因为这深刻受到解析器（如获取不同多模态或表格识别）、切片大小（Chunk Size）、重叠区域（Chunk Overlap）、切片算法逻辑以及所采用的 Embedding 模型的直接影响。这要求系统底层配置必须进行固化以保证检索的精确复现。

### 2. Chunk 的精准提取与 API 返回支持
在 RAG 的追根溯源中，AnythingLLM 完全支持将查找到的 Chunk 返回给业务后端。
- 当向其 `/api/v1/workspace/{slug}/chat` 接口发起检索请求时，返回的 JSON 响应不仅包含大模型的标准回答（`textResponse`），还在同级的 `sources` 数组中原封不动地返回检索到的所有源片段（Chunk）。
- 其中包含了对应的原始内容文本（`text`）、置信度/相似度得分（`score`）以及原始文件名（`title`），完美满足了对“溯源证据链”的精确返回要求。

### 3. 文件的存储路线与安全性
- **全局库与 Workspace 分离**：AnythingLLM 把“上传全局库”和“将文件融入特定工作区（Workspace）知识库”做成了两阶段解耦的操作。
- 当文件经由 `/document/upload` 进入系统后，其物理文件和目录名称会被进行安全化处理（Slugify，去除空格特殊符号防范路径解析错误），同时在其物理路径同级生成一份标准的结构化提取数据（如 `custom-documents/xxx-idxxx.json`）。这份包含纯文本内容和元数据的 JSON 才是真正的向量生成底座，不仅能在底层数据库中对应回真实原文件名，也能支持极其廉价且健壮的掉电重建工作。

### 4. Workspace (工作区) 宏观路由与分类设计
在面对海量极其相似的文本时，将所有资料混在一个“超级全局库”里盲搜是严重的系统痛点且极易引起大模型的幻觉提取（“张冠李戴”）。
- 为此系统采用动态路由思想，避免建立全库检索，而是原生强制采用**每个类别映射为一个 AnythingLLM 的 Workspace（且以唯一 slug 作为标识）**的设计方案。
- 在后续检索前，利用这层物理宏观屏障，使得对于如某件装备或某一类别的需求能百分百瞬间压缩在属于自己的绝佳知识空间内进行精细提纯。

### 5. Metadata (元数据) 的微观过滤支持
在 Workspace 隔离的内部，更进阶的搜索防伪手段是针对各个 Chunk 上传并打上所属的元标签（Metadata）。
- 当将新文件命令为向量并“嵌入工作区”时，可通过相应的 `/document/meta` 功能将与它强关联的需求属性（如 `{"file_name":"xx", "country":"美国", "channel":"装发"}` 等多维条件）伴随着请求透传给底层系统。
- 最终 AnythingLLM 会把这笔字典里的标量属性在向量数据库中与这几百个 Chunk 牢牢绑定在同列。后续若针对这些字段进行查询时，数据库会启用传统标量过滤（类似于 SQL 的 `WHERE` ），瞬间剔除无用文本，进一步提升查找效率和防污能力。

## 二、 SQLite 中单节点存表的控制面应用机制

针对于后端的调度与维护，单纯将所有的数据都挂靠给 AnythingLLM 的查询显然会带来管理层面的缓慢与高门槛。为确保大批量甚至异步任务对于这些分类和文档存储的操作健壮不互相冲突，基于 SQLite 构建了隔离的业务表设计。

### 1. 锁池设计与代码封装机制
- 绝不在 Flask 的外部视图层直接编写 SQL。所有的数据库连接被封装在了专用的 Python 任务类中（如 `DatabaseService`）。
- 因为涉及到多线程在处理同一类文件的长期文档处理异步流转，系统内置了 Python `threading.Lock()` （并发读写互斥锁）以及极简的连接会话生命期，防范死锁并严守并发安全底线。

### 2. 精简的数据库表态双核路线设计
将表的设计彻底减负，不储存无需复盘的大段文本向量和杂乱信息，做到了只存核心路由指纹：
- **`workspaces`分类映射表设计**：用于管理宏观的 “建筑架构树（类别 ID）” 走去哪。其中仅存储 `id`（自身主键）、业务请求来的 `architecture_id`（类别）、和向 AnythingLLM 通信时的那个 `workspace_slug` 标识。此结构允许极其轻易地以 `O(1)` 完成类别分发的寻址。
- **`documents`明细留存表设计**：用于单点管理存续文件，做定点删除溯源。表中仅储存业务的 `file_name`（文档名主键），它挂靠的类别 `architecture_id`，以及极为重要的 AnythingLLM 中给它签发的物理底层标识 `anything_doc_id`。这就使得无论是对库状态容灾对账，或是未来对某个单文件的精准查删和防改动均游刃有余。

## 三、 本次分类检索大库整合开发的设计落成总结

在对上面两大引擎原理充分吃透后，代码逻辑成功地对一次正常的业务上传走完了“从大模型提取结构化，落到分类向量仓，双轨登记在列”的完整长链路调度（位于系统的 `llm_analysis_service.py` 环节中）。

### 1. RAG与持久化处理的安全流转与隔离双写闭环：
- 在调用大模型读取、理解、映射并执行完结构化的操作（确信已**成功完成前置解析**不引发提前报错）后，系统切入持久化步骤。
- **第一步：自动扩容与惰性初始（Lazy Initialization）新建类别墙。**依据提取到的 `architectureId`，向 SQLite 大表要工作区 `slug`，若没要到，说明这是崭新的分类系统；后台直接利用 API 自主在 AnythingLLM 开辟新工作区，并立即回写进 SQLite 的 `workspaces` 落库备存以服务后续兄弟文件的进入。
- **第二步：独立传存并烙印 Metadata 提供微观精准保护。**根据工作区去向明确后将原文交由原生客户端（`upload_document`）产生底层的 json 文档源标识，配合解析提取出来的包括国家、渠道、甚至 `architecture_id` 等诸多元数据构成一个专门的 JSON 体，通过封装后的透传机制在绑定工作区的一瞬间全量发送给 `/document/meta` 给切好成段的 Chunks 各自打上标签。
- **第三步：异常捕获防御。**这一切与底层 API 进行通讯期间（如获取不到原物理 `doc_relative_path` 路径漂移时），系统立即采用了诸如构造包含原生 `{doc_id}.json` 为名称的 `alt_path` 做第二次重试上传等严密的容错退路。
- **第四步：控制面双写封口。**在完全安全地确定文件已经在 AnythingLLM 内生根发芽后，代码立即触发 `kb_service.save_document_record`，将上述成功得到的这一大捆指纹锁入了 SQLite，确保后续一切改配不丢失节点；全程并置放在了完备的异常外衣内以不打扰后续的主业务进度条与回调执行。
