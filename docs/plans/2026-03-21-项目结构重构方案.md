# Flask App 目录重构建议方案

经过对 `app/` 下目录的深入分析，目前的问题主要集中在：所有相关业务、外部客户端调用、工具函数、Prompt 模板全部堆砌在 `app/services/` 中（总计近 20 个文件或包），缺乏层级隔离，随着项目发展会越来越难以维护。

下面为您提供一个**分层清晰、职能单一**的重构组织结构。由于所有的文件迁移和代码中 `import` 路径的修改将由您**手动完成**，这里只梳理最佳的摆放位置：

---

## 现在的 `app/` 内部目录体系 (已按照最新的大重构落地)

本次重构非常彻底，将所有的基础配置、外部客户端、工具类、核心大模型服务等，都集中收拢进了 `app/services/` 下进行内部分层，使得对外的暴露面极小，内聚性极强！

### 1. `app/blueprints/` —— 路由控制层
作为对外的唯一交互层，内部不再掺杂任何实现细节。
* `llm.py` (处理 `/llm/` 前缀的 HTTP 控制器)
* `__init__.py`

### 2. `app/services/` —— 核心封装层 (重构重点)
被拆分为了四个职能高度专一的子模块层：

#### 2.1 `core/` —— 核心配置与基础组件
专门存放不包含具体业务流程的“原子级”设置和全应用通用组件：
* `config.py` (加载大模型、OCR等环境变量配置)
* `settings.py` (存放静态目录常量，如 `UPLOAD_DIR`)
* `logging.py` (日志系统配置)
* `database.py` (轻量级的本地 sqlite 数据库连接和记录读写抽象)
* `progress_hub.py` (基于 WebSocket 的全局跨模块进度分发器)
* `prompts.py` (所有 LLM 操作的 Prompt 词的统一归口，告别散落各处的硬编码)

#### 2.2 `llm_service/` —— 核心业务逻辑
去除了杂乱名称，专注具体领域的数据加工和业务编排（“引擎”部分）：
* `analysis_service.py` (通用文件层面的解析业务流)
* `report_service.py` (多文件到报告的聚合生成业务流)
* `weaponry_service.py` (专注于知识谱系领域的武器字段精确提取)
* `task_service.py` (统筹所有异步解析任务的派发、状态扭转和状态追踪)
* `translation_service.py` (大模型与本地翻译的协同策略封装)

#### 2.3 `utils/` —— 客户端、工具箱与底层执行器
纯粹的方法依赖层和外部服务 SDK 层：
* `anythingllm_client.py` (专门屏蔽 AnythingLLM 底层 HTTP 请求细节的第三方 SDK 封装)
* `callback_client.py` (统一的 HTTP 回调发送工具)
* `rag_pipeline.py` (最终将文件和 Prompt 拼在一起提交给 LLM 的流水线调用逻辑)
* `ocr_preprocessor.py` (整合了原先 `document_utils.py` 中的扫描件和 PDF 判定逻辑，加上文字提取核心逻辑)
* `mhtml_normalizer.py` (对混入特定乱码和嵌套标签格式的网页文件进行标准化过滤)
* `file_downloader.py` (纯粹负责基于本地/远程 URL 的文件拉取保存)

#### 2.4 `hy_mt_translator/`
* 本地的定制化离线或者专用外部翻译服务算法包。

---

## 🎯 各层调用规范 (防依赖环)
* **Blueprints (Controller)** 负责接收 HTTP 请求参数拦截校验，调用 **Workflows** 或 **Services**。不直接写业务。
* **Workflows** 负责串联多个 **Services** （例如先调用下载，再传给翻译，再传给大模型解析）。
* **Services** 拼装业务逻辑，获取 **配置/Prompts**（从 Core 引入），调用 **Clients** 拿数据，或者通过 **Utils** 处理数据。需要读写库时调用 **Core/DB**，需要推送进度时调用 **Progress Hub**。
* **Utils** 最底层，不依赖任何自身项目的业务组件，纯 Python 处理函数。
* **Clients** 只负责跟外部打交道，做 HTTP Request 和 Response 封装。

## 重构步骤建议：
1. **建立文件夹**：参照上面的列表，在 `app/` 内建立 `clients`, `workflows`, `utils`, `core`（不再需要 `prompts/` 文件夹）。
2. **合并与移动文件**：将 `services/llm_prompts.py` 和 `services/llm_weaponry_prompts.py` 两者内容合并复制到一个新建的文件 `app/core/prompts.py` 中，然后再把其他杂乱的文件分门别类拖入对应的文件夹。
3. **重命名引用（批量替换）**：使用 IDE(VSCode/PyCharm) 的全局搜索功能，把原来的绝对引用进行批量替换。
4. **运行单元测试**：全部修改并无语法报错后运行您的 `pytest` 检测是否有导包遗留问题。

这套架构属于比较标准的职责驱动包目录设计（DDD/Clean Architecture 变体），将有效支撑您的甲方项目后续长期迭代。您现在可以照此思路手动移动文件了，如果有任何局部设计疑问您可以随时发问。
