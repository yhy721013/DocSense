# Flask App 目录重构建议方案

经过对 `app/` 下目录的深入分析，目前的问题主要集中在：所有相关业务、外部客户端调用、工具函数、Prompt 模板全部堆砌在 `app/services/` 中（总计近 20 个文件或包），缺乏层级隔离，随着项目发展会越来越难以维护。

下面为您提供一个**分层清晰、职能单一**的重构组织结构。由于所有的文件迁移和代码中 `import` 路径的修改将由您**手动完成**，这里只梳理最佳的摆放位置：

---

## 推荐的 `app/` 内部目录体系

### 1. `app/core/` —— 核心配置与基础组件
专门存放应用级别的基础和核心配置组件，不包含具体业务逻辑。
* `config.py` (由根目录移入，包含 `load_anythingllm_config` 等基础配置加载)
* `settings.py` (全局静态常量，如 `UPLOAD_DIR` )
* `logging_config.py`
* `database_service.py` (数据库连接与基础操作剥离放置此处或建一个 `app/database/` 专门处理连接)
* `llm_progress_hub.py` (基于 WebSocket 的全局进度分发组件)

### 2. `app/clients/` (或 `app/integrations/`) —— 第三方交互包装
所有向外发起请求的 SDK 和客户端。剥离它们能够有效解耦外部依赖。
* `anythingllm_client.py` (AnythingLLM 专用客户端)
* `llm_callback_service.py` (如果它只是一个负责往外部发送 HTTP JSON Callback 的工具组件，建议重命名为 `callback_client.py` 并放入此处)

### 3. `app/prompts/` —— 大模型提示词统一管理
现在的 Prompt 直接嵌在代码或者零散文件里，抽离出来更方便调优。
* `llm_prompts.py`
* `llm_weaponry_prompts.py`

### 4. `app/services/` —— 纯净的业务逻辑服务层 (按领域拆分)
原来的 `app/services/` 过于臃肿，里面的服务层应仅保留“大脑”部分，即处理业务的数据流：
* **核心理解/解析业务**：
  * `llm_analysis_service.py` (文件解析处理编排)
  * `llm_report_service.py` (报告生成编排)
  * `llm_weaponry_service.py` (知识谱系/武器相关的解析编排)
  * `llm_task_service.py` (大模型异步任务状态的轮询流转检查控制)
* **翻译包与服务**：
  * `llm_translation_service.py`
  * `hy_mt_translator/` (翻译用的依赖包文件夹)

### 5. `app/workflows/` (或 `app/pipelines/`) —— 流水线引擎
这里用于存放那些将多个 Service、Client、Utils “串联”起来的顶层执行流，也就是 Controller(Blueprint) 到 Service 之间的胶水层编排：
* `pipeline.py`
* `rag_with_ocr.py` 

### 6. `app/utils/` —— 通用工具包
职责必须很“纯”，只做一件事（如格式化、转换、下载），没有项目相关强业务概念。
* `document_utils.py` (根目录的文档装载工具)
* `mhtml_normalizer.py` (专门处理 mhtml 格式过滤)
* `ocr_preprocessor.py` (图片提取文字工具)
* `llm_download_service.py` (文件下载实用函数，纯工具，可考虑更名为 `file_downloader.py`)

### 7. `app/blueprints/` —— 路由控制层
* `llm.py` (目前有 17KB，建议后期如果变大，可以按照 `/llm/analysis`、`/llm/weaponry` 拆分为多个 Controller 文件。)
* `__init__.py`

---

## 🎯 各层调用规范 (防依赖环)
* **Blueprints (Controller)** 负责接收 HTTP 请求参数拦截校验，调用 **Workflows** 或 **Services**。不直接写业务。
* **Workflows** 负责串联多个 **Services** （例如先调用下载，再传给翻译，再传给大模型解析）。
* **Services** 拼装业务逻辑，获取 **Prompts**，调用 **Clients** 拿数据，或者通过 **Utils** 处理数据。需要读写库时调用 **Core/DB**，需要推送进度时调用 **Progress Hub**。
* **Utils** 最底层，不依赖任何自身项目的业务组件，纯 Python 处理函数。
* **Clients** 只负责跟外部打交道，做 HTTP Request 和 Response 封装。

## 重构步骤建议：
1. **建立文件夹**：参照上面的列表，在 `app/` 内建立 `clients`, `prompts`, `workflows`, `utils`, `core`。
2. **分类移动**：把 `services` 里杂乱的文件分门别类拖入对应的文件夹。
3. **重命名引用（批量替换）**：使用 IDE(VSCode/PyCharm) 的全局搜索功能，把诸如 `from app.services.llm_prompts` 换为 `from app.prompts.llm_prompts` 等等。
4. **运行单元测试**：全部修改并无语法报错后运行您的 `pytest` 检测是否有导包遗留问题。

这套架构属于比较标准的职责驱动包目录设计（DDD/Clean Architecture 变体），将有效支撑您的甲方项目后续长期迭代。您现在可以照此思路手动移动文件了，如果有任何局部设计疑问您可以随时发问。
