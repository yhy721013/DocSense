# 文件对话 — 后端内部设计文档

> 此文档为后端实现参考，不对甲方暴露。

## 1. AnythingLLM 资源映射策略

### 隔离模型：1 对话 = 1 Workspace + 1 Thread

| 甲方概念 | 后端概念 | AnythingLLM 资源 |
|----------|----------|------------------|
| chatId | 对话记录 | 1 个独立 Workspace + 1 个 Thread |
| fileNames | 引用文件 | Workspace 级别的 document embeddings |

**选择此策略的原因**：AnythingLLM 的 RAG 文档嵌入是 Workspace 级别的，同一 Workspace 下所有 Thread 共享 embeddings。为避免不同对话引用不同文件时互相干扰，每个 chatId 独占一个 Workspace。

### 引用文件变更处理

当同一 chatId 后续请求的 `fileNames` 发生变化时：

```text
老文件集合: {A, B}
新文件集合: {B, C}
  → 需要添加: {C}  →  upload + update-embeddings (adds)
  → 需要移除: {A}  →  update-embeddings (deletes)
  → 保持不变: {B}  →  无操作
```

通过集合差运算（`new - old` / `old - new`）计算增量，调用 `update-embeddings` 的 `adds` / `deletes` 完成更新。

## 2. 数据库设计

### 对话记录表 `chats`

存储于 `knowledge_base.sqlite3`（复用现有 `DatabaseService`）。

| 字段 | 类型 | 说明 |
|------|------|------|
| chat_id | TEXT PK | 对话唯一标识（甲方传入） |
| file_names | TEXT | 当前引用的文件列表（JSON 数组） |
| workspace_slug | TEXT | 对应 AnythingLLM workspace slug |
| thread_slug | TEXT | 对应 AnythingLLM thread slug |
| created_at | TEXT | 创建时间（ISO 8601） |
| updated_at | TEXT | 最近更新时间 |

## 3. 接口处理流程

### POST `/llm/chat` — 发送对话消息（SSE 流式）

```text
收到请求
  → 校验 chatId / fileNames / message 非空
  → 校验 fileNames 中的文件均已解析（查 documents 表）
  → 校验失败 → 直接返回 JSON 错误（400/404），不建立 SSE 连接
  → 校验通过 → 建立 SSE 连接（Content-Type: text/event-stream）
  → 查询 chats 表
    → 不存在（新对话）:
      1. 创建 Workspace（名称: chat-{chatId}）
      2. 将 fileNames 对应的文档嵌入 Workspace（upload + update-embeddings）
      3. 创建 Thread
      4. 写入 chats 表
      5. isNewChat = true
    → 已存在（继续对话）:
      1. 对比 fileNames 与数据库中的 file_names
        → 有变更: 增量 update-embeddings → 更新 chats 表的 file_names
        → 无变更: 跳过
      2. isNewChat = false
  → 推送 chatInfo 事件: { chatId, isNewChat }
  → 向 Thread 发送 message（mode=chat, 流式）
  → 逐片段推送 textChunk 事件: { content }
  → 推送 done 事件: { chatId }
  → 运行时错误 → 推送 error 事件后关闭连接
```

### GET `/llm/chat/history` — 获取对话历史

```text
收到请求
  → 校验 chatId 非空（Query 参数）
  → 查询 chats 表
    → 不存在 → 返回 404
    → 存在:
      1. 通过 workspace_slug + thread_slug 调用 AnythingLLM Thread Chat API
         获取该 Thread 下的所有历史消息
      2. 将消息转换为 { role, content } 格式的列表（按时间正序）
      3. 从 chats 表读取当前 file_names
      → 返回 200: { chatId, fileNames, messages }
```

### POST `/llm/chat/delete` — 删除对话

```text
收到请求
  → 校验 chatId 非空
  → 查询 chats 表
    → 不存在 → 返回 404
    → 存在:
      1. 删除 AnythingLLM Thread（尽力删除，失败不阻塞）
      2. 删除 AnythingLLM Workspace（1:1 隔离，可安全删除）
      3. 删除 chats 表记录
      → 返回 200
```

## 5. 关键设计决策

### SSE 流式输出

对话接口采用 **SSE（Server-Sent Events）流式推送**。原因：
- 大模型生成回答时需要较长时间，流式输出可以让用户感知到实时响应，显著提升体验
- AnythingLLM 的 chat API 本身支持流式响应，不做流式转发会白白增加用户等待感
- SSE 基于标准 HTTP，客户端无需 WebSocket 即可接入，与现有项目无冲突

**Flask 实现要点**：使用 `Response(generator(), mimetype='text/event-stream')` 配合生成器函数逐片段推送。需设置 `Cache-Control: no-cache` 和 `X-Accel-Buffering: no` 以禁用代理缓冲。

### 对话历史从 AnythingLLM 获取

- AnythingLLM Thread 内部已完整维护对话上下文（每轮 user 消息和 assistant 回答）
- `/llm/chat/history` 接口直接从 AnythingLLM 读取 Thread 历史，无需在 DocSense 侧冗余存储
- 减少存储负担且保证历史与实际上下文严格一致
- **潜在风险**：若 AnythingLLM 数据丢失（如重装），历史将不可恢复。但因文件嵌入也会丢失，此时对话已无法正常继续，需要新建对话，所以影响有限。

### 对话列表由甲方维护

- 甲方创建对话（生成 chatId）和删除对话都会通知我方
- 甲方自然拥有完整的对话列表
- 我方不掌握用户体系信息，提供列表接口意义不大
