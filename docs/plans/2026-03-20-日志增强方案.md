# 综合日志增强方案

项目目前的日志输出较为零散且不统一。本方案旨在建立一套标准化的日志规范，覆盖请求入口、后台任务及外部接口调用，以提升系统的可维护性和排障效率。

## 拟提出的更改

### 1. 基础配置
#### [新增] [logging_config.py](file:///c:/Files/Codes/DocSense/app/logging_config.py)
- 配置统一的日志格式：`[%(asctime)s] %(levelname)s [%(name)s:%(lineno)d] %(message)s`。
- 设置默认日志级别为 `INFO`。

#### [修改] [__init__.py](file:///c:/Files/Codes/DocSense/app/__init__.py)
- 在 `create_app` 入口处初始化日志配置。

### 2. 外部调用增强
#### [修改] [anythingllm_client.py](file:///c:/Files/Codes/DocSense/anythingllm_client.py)
- 在 API 调用失败（非 200 响应、网络异常等）时增加详细的错误日志。
- 对关键操作（如创建 Workspace、Thread、更新 Embeddings）增加执行日志。

### 3. 服务层增强 (app/services/)
- 将所有 `print(...)` 语句替换为标准的 `logger` 调用（`info`, `warning`, `error`）。
- 在各服务的核心函数（如 `run_weaponry_task`, `run_file_analysis_task`, `translate_document` 等）的起始和完成处增加日志记录。
- 记录关键的业务状态转变和异常详情。

### 4. 路由层增强 (app/blueprints/)
- 在 `chat.py`, `classify.py`, `llm.py` 等 Blueprint 中，对关键接口的请求参数。
- 对后台异步任务的创建和启动增加日志记录。

## 验证计划

### 自动化验证
- 启动应用并监听控制台输出，触发各类业务流程（分类、对话、武器装备、报告生成）。
- 确认日志格式符合预期，能够清晰展现任务的生命周期和异常原因。

### 手动验证
- 模拟 AnythingLLM 异常（如关闭服务），确认系统能准确记录详细的 HTTP 错误和排查信息。
