# 文件对话调试页体验优化设计

## 1. 背景

DocSense 当前已经具备基础的 `/debug/chat` 本地调试页，能够完成以下能力：

1. 查看本地会话列表
2. 选择已解析文件
3. 发起 `/llm/chat` SSE 流式请求
4. 查询 `/llm/chat/history`
5. 删除会话

当前实现已经满足功能闭环，但在交互和布局层面存在明显问题：

1. `已解析文件` 仍使用原生多选框，已选状态与取消选择的表达不直观。
2. 右侧页面仍是“字段表单 + 聊天记录 + SSE 事件流”三段式调试布局，不够接近常见大模型 web 页面。
3. SSE 虽然已支持流式逻辑，但主界面信息层次仍偏技术视角，不够符合良好用户体验。
4. 页面字段与聊天结果分散在多个卡片中，用户对当前上下文、已选文件、流式回复状态的感知不够集中。

因此，本期目标不是补新接口，而是在不修改真实 chat 协议、不影响甲方回调链路的前提下，对 `/debug/chat` 做一轮前端交互与布局优化。

## 2. 目标

本期需要完成以下目标：

1. 将 `已解析文件` 从原生多选框升级为更直观的勾选式文件选择器。
2. 在文件候选列表中为每个文件名提供 checkbox，明确展示是否选中。
3. 支持用户取消勾选文件，并与“已选文件”展示保持同步。
4. 保持左侧“本地会话列表”基本不变。
5. 将右侧原本分离的“聊天”框与“聊天记录”框整合为一个主聊天区域。
6. 右侧整体布局更接近常见大模型 web 页面：
   - 顶部是会话与上下文控制
   - 中部是消息流
   - 底部是输入区
7. SSE 流式输出在主聊天区中实时展示模型回复，而不是仅在技术事件区中体现。
8. 调试信息仍然保留，但折叠到次级信息层，不抢主视觉。
9. 所有优化必须继续与甲方真实回调链路隔离。

## 3. 非目标

以下内容明确不在本期范围内：

1. 不修改 `/llm/chat`、`/llm/chat/history`、`/llm/chat/delete` 的协议。
2. 不修改 `app/services/llm_service/chat_service.py` 的业务逻辑。
3. 不修改 `AnythingLLM` 的线程、工作区、历史读取或删除策略。
4. 不让 `/debug/chat` 写入 `.runtime/call_back.json`。
5. 不引入 React、Vue、Vite、Next.js 等独立前端工程。
6. 不增加聊天搜索、消息编辑、消息重试、附件上传、消息点赞等产品功能。
7. 不改变左侧“本地会话列表”的核心职责和数据来源。

## 4. 约束与前提

1. 当前 `/debug/chat` 仍基于 Flask 模板 + 原生 JavaScript。
2. 页面初始化数据仍然来自 `GET /debug/api/chat/bootstrap`。
3. 消息发送、历史查询、删除会话仍直接调用正式接口。
4. 左侧会话列表仍以本地 `chat_sessions.sqlite3` 为数据来源。
5. 文件候选列表仍以本地 `knowledge_base.sqlite3.documents` 为数据来源。
6. 优化必须保证“仅改善本地调试体验”，不能影响甲方真实回调服务。
7. 以下硬约束保持不变：
   - 不修改 `callback_client.py`
   - 不修改 `callback_preview.py`
   - 不修改 `/debug/callback` 与 `/debug/api/callback`
   - 不新增伪回调或额外回调写盘

## 5. 方案选择

### 5.1 备选方案

#### 方案 A：轻量重组

保留现有多卡片结构，只把文件选择器换成勾选列表，并将 SSE 区改为折叠区。

#### 方案 B：聊天产品化改版

保留左侧本地会话列表，右侧整体改造成更接近常见大模型页面的单一聊天主卡片；文件选择器改为“已选标签 + 添加文件面板”；SSE 在主消息区实时展示，同时保留折叠调试详情。

#### 方案 C：极简增强

仅替换文件多选控件，不调整右侧布局结构。

### 5.2 结论

本期采用方案 B。

原因如下：

1. 该方案最符合“更贴近常见大模型 web 页面”的目标。
2. 可以同时解决文件选择可读性、聊天主视图分裂、SSE 主体验不足三个问题。
3. 右侧统一为一个聊天主区域后，用户心智更清晰。
4. 该方案仍然只改模板与前端交互，不触碰真实 chat 协议与回调链路。

## 7. 页面信息架构

### 7.1 左侧：本地会话列表

左侧区域保留现有职责，不做本期重点改动。

保留以下内容：

1. 标题“本地会话列表”
2. 本地会话卡片列表
3. 点击切换会话并加载历史

仅允许做轻微视觉对齐，不改变交互模型。

### 7.2 右侧：统一聊天主区域

右侧改造成单一主卡片，而不是多个并列/串行卡片。

该主卡片从上到下分成四层：

1. 会话控制栏
2. 文件上下文栏
3. 消息流区域
4. 底部输入区

#### 7.2.1 会话控制栏

展示：

1. `chatId` 标签与输入框
2. `加载历史` 按钮
3. `删除当前会话` 按钮

设计原则：

1. `chatId` 明确可见，不隐藏到高级面板
2. `chatId` 仍允许手工编辑
3. 控制按钮与 `chatId` 同处一层，强调当前会话上下文

#### 7.2.2 文件上下文栏

展示：

1. 已选文件标签区
2. `添加文件` 按钮
3. 可展开的文件选择面板

已选标签规则：

1. 每个已选文件显示为独立标签
2. 标签上可显示：
   - 文件名
   - 可选的次级信息 `architectureId`
3. 每个标签带一个移除入口，用于取消选择

#### 7.2.3 消息流区域

这是右侧视觉中心。

展示：

1. 历史 `user` 消息
2. 历史 `assistant` 消息
3. 当前 SSE 流中的 assistant 回复

设计原则：

1. 用户消息与模型消息采用不同气泡样式
2. 按时间顺序自上而下排列
3. 当前流式输出在 assistant 气泡中实时增长
4. 不再把 SSE 主输出放在单独技术面板中

#### 7.2.4 底部输入区

展示：

1. `message` 输入框
2. `发送消息` 主按钮
3. 调试详情折叠入口

设计原则：

1. 输入区始终位于聊天主区域底部
2. `发送消息` 是主视觉按钮
3. 调试信息是次级入口，不抢主交互

## 8. 文件选择器设计

### 8.1 候选文件面板

点击 `添加文件` 后，展开一个可滚动面板。

面板内每一项展示：

1. checkbox
2. 文件名
3. `architectureId` 次级信息

交互规则：

1. 勾选即加入已选文件集合
2. 取消勾选即从已选文件集合移除
3. 面板关闭后选择结果保留
4. 再次打开时应保留当前勾选状态

### 8.2 已选文件标签区

标签区是页面的主可见状态，不要求用户频繁打开候选面板才能知道当前上下文。

规则：

1. 已选文件必须始终可见
2. 标签移除动作应与勾选面板状态同步
3. 当没有已选文件时，显示空状态提示

### 8.3 单一真相源

页面内部最终仍以一组 `selectedFileNames` 作为唯一真相源。

以下视图都从该状态派生：

1. 已选标签区
2. 候选面板勾选状态
3. 发送 `/llm/chat` 时的 `fileNames`

## 9. SSE 展示与调试信息设计

### 9.1 主聊天区流式输出

SSE 的主体验从“技术事件列表”切换为“聊天气泡实时输出”。

规则：

1. 收到 `chatInfo` 时，更新当前会话元信息
2. 收到 `textChunk` 时，将内容持续追加到当前 assistant 气泡
3. 收到 `done` 时，将该 assistant 回复固化为历史消息
4. 收到 `error` 时，停止增长并在主界面提示错误

### 9.2 调试信息折叠区

SSE 调试信息仍然保留，但默认折叠。

展开后可查看：

1. `chatInfo`
2. `textChunk`
3. `done`
4. `error`

展示要求：

1. 保持原始事件顺序
2. 可查看 JSON 结构
3. 出错时折叠区需有明显错误标记

### 9.3 视觉优先级

优先级从高到低为：

1. 当前聊天内容
2. 已选文件上下文
3. 会话控制动作
4. 调试细节

这样既满足联调需求，也更符合常见大模型页面的阅读路径。

## 10. 状态管理设计

本期不改变总体状态模型，但会做职责重组。

### 10.1 `activeChatState`

负责：

1. 当前 `chatId`
2. 当前会话是否存在
3. 当前会话相关按钮可用性

### 10.2 `selectedFilesState`

新增或强化独立职责：

1. 当前选中的文件名集合
2. 已选标签区渲染
3. 候选面板勾选状态
4. 标签移除动作同步

### 10.3 `historyState`

继续负责：

1. 当前历史消息列表
2. 历史加载状态

### 10.4 `streamState`

明确拆分为：

1. `streamingReplyText`
2. `streamEvents`
3. `streamStatus`
4. `streamError`

这样主聊天区和调试详情区可以共享同一轮流式数据，而不互相耦合。

## 11. 异常处理

### 11.1 表单前置校验

以下场景阻止发送：

1. `chatId` 为空
2. 未选择任何文件
3. `message` 为空

提示位置应尽量靠近当前操作区，而不是只在全局 banner 中显示。

### 11.2 流式期间限制

当 SSE 正在进行时：

1. 禁止再次发送
2. 禁止切换左侧会话
3. 禁止删除当前会话
4. 禁止触发新的历史加载

### 11.3 SSE 失败

当收到 `error` 事件时：

1. 主聊天区停止继续输出
2. 当前 assistant 气泡标记为失败
3. 调试详情区高亮错误状态
4. 不影响 callback 链路，不写 callback 文件

3. 已选文件能以标签形式持续可见。
