## 1. 发送对话消息接口（SSE 流式）

**接口路径：** `/llm/chat`  
**请求方法：** `POST`  
**响应方式：** SSE（Server-Sent Events）流式输出  
**Content-Type：** `text/event-stream`  
**功能说明：** 基于指定文件内容，向大模型发送用户问题并以 SSE 流式返回回答。若对话 ID 不存在则自动创建对话；若传入新增文件则自动追加引用。

### 请求参数说明

| 参数 | 类型 | 说明 | 必填 |
|------|------|------|------|
| businessType | String | 业务类型标识，固定值：`"chat"` | 是 |
| params | Object | 对话参数（非数组） | 是 |

### params 参数结构

| 字段 | 类型 | 说明 | 必填 | 示例值 |
|------|------|------|------|--------|
| chatId | String | 对话唯一标识（由调用方生成） | 是 | `"conv-20260401-001"` |
| fileNames | Array\<String\> | 本次新增引用的文件列表（增量，已解析过的文件）。新对话时必须非空；继续对话时可为空数组 `[]` 表示不引用新文件 | 是 | `["abc123.pdf"]` |
| message | String | 用户发送的问题内容 | 是 | `"请总结这份文件的核心观点"` |

> 调用方无需传入此前的对话历史消息，后端自动维护对话上下文的连贯性。

### 请求示例

```json
{
  "businessType": "chat",
  "params": {
    "chatId": "conv-20260401-001",
    "fileNames": [
      "abc123def456.pdf",
      "789ghi012jkl.docx"
    ],
    "message": "请总结这两份文件的核心观点"
  }
}
```

### SSE 事件流说明

连接成功后，服务端以 `text/event-stream` 格式逐步推送以下事件：

#### 事件类型

| event | data 内容 | 说明 |
|-------|-----------|------|
| `chatInfo` | `{"chatId": "...", "isNewChat": true/false}` | 首个事件，返回对话元信息 |
| `textChunk` | `{"content": "..."}` | 大模型回答的文本片段，可能多次推送 |
| `done` | `{"chatId": "..."}` | 流式输出完成 |
| `error` | `{"error": "..."}` | 出错时推送，推送后关闭连接 |

#### SSE 流示例

```text
event: chatInfo
data: {"chatId": "conv-20260401-001", "isNewChat": true}

event: textChunk
data: {"content": "根据文件内容，"}

event: textChunk
data: {"content": "核心观点包括："}

event: textChunk
data: {"content": "1. ......"}

event: done
data: {"chatId": "conv-20260401-001"}
```

### 错误响应

参数校验等在建立 SSE 连接前发生的错误，仍以普通 JSON 响应返回：

#### 参数错误（HTTP 400）

```json
{
  "error": "新对话的fileNames不能为空"
}
```

#### 引用文件未解析（HTTP 404）

```json
{
  "error": "文件 abc123.pdf 尚未解析，无法用于对话"
}
```

SSE 连接建立后的运行时错误，通过 `error` 事件推送：

```text
event: error
data: {"error": "大模型服务响应异常"}
```

## 2. 获取对话历史接口

**接口路径：** `/llm/chat/history`  
**请求方法：** `GET`  
**功能说明：** 根据 `chatId` 获取指定对话的历史消息记录。

### 请求参数说明

| 参数 | 类型 | 传递方式 | 说明 | 必填 |
|------|------|----------|------|------|
| chatId | String | Query 参数 | 对话唯一标识 | 是 |

### 请求示例

```
GET /llm/chat/history?chatId=conv-20260401-001
```

### 响应参数说明

| 字段 | 类型 | 说明 |
|------|------|------|
| chatId | String | 对话唯一标识 |
| fileNames | Array\<String\> | 当前引用的文件列表 |
| messages | Array\<Object\> | 对话历史消息列表（按时间正序） |

### messages 数组元素结构

| 字段 | 类型 | 说明 |
|------|------|------|
| role | String | 消息角色：`"user"` 或 `"assistant"` |
| content | String | 消息内容 |

### 响应示例

#### 成功响应（HTTP 200）

```json
{
  "chatId": "conv-20260401-001",
  "fileNames": ["abc123def456.pdf", "789ghi012jkl.docx"],
  "messages": [
    {
      "role": "user",
      "content": "请总结这两份文件的核心观点"
    },
    {
      "role": "assistant",
      "content": "根据文件内容，核心观点包括：1. ...... 2. ......"
    }
  ]
}
```

#### 参数缺失（HTTP 400）

```json
{
  "error": "chatId不能为空"
}
```

#### 对话不存在（HTTP 404）

```json
{
  "error": "对话不存在"
}
```

## 3. 删除对话接口

**接口路径：** `/llm/chat/delete`  
**请求方法：** `POST`  
**功能说明：** 删除指定对话，同步清理后端资源

### 请求参数说明

| 参数 | 类型 | 说明 | 必填 |
|------|------|------|------|
| businessType | String | 固定值：`"chat"` | 是 |
| params | Object | 删除参数 | 是 |

### params 参数结构

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| chatId | String | 要删除的对话唯一标识 | 是 |

### 请求示例

```json
{
  "businessType": "chat",
  "params": {
    "chatId": "conv-20260401-001"
  }
}
```

### 响应示例

#### 删除成功（HTTP 200）

```json
{
  "chatId": "conv-20260401-001",
  "deleted": true,
  "msg": "对话已删除"
}
```

#### 对话不存在（HTTP 404）

```json
{
  "error": "对话不存在"
}
```

## 4. 调用方职责说明

以下数据与功能由调用方自行维护。

### 4.1 对话列表管理

调用方应在自身系统中维护对话列表，记录 `chatId`（对话唯一标识）、创建者等信息。

当调用方不再需要某对话时，应调用 `/llm/chat/delete` 通知后端同步释放资源。

### 4.2 对话历史

调用方可通过 `/llm/chat/history` 接口随时获取指定对话的完整历史消息，用于界面展示和回溯。
