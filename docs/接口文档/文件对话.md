## 1. 发送对话消息接口

**接口路径：** `/llm/chat`  
**请求方法：** `POST`  
**功能说明：** 基于指定文件内容，向大模型发送用户问题并获取回答。若对话 ID 不存在则自动创建对话；若引用文件列表与上次不同则自动更新引用。

### 请求参数说明

| 参数 | 类型 | 说明 | 必填 |
|------|------|------|------|
| businessType | String | 业务类型标识，固定值：`"chat"` | 是 |
| params | Object | 对话参数（非数组） | 是 |

### params 参数结构

| 字段 | 类型 | 说明 | 必填 | 示例值 |
|------|------|------|------|--------|
| chatId | String | 对话唯一标识（由调用方生成） | 是 | `"conv-20260401-001"` |
| fileNames | Array\<String\> | 引用的文件对象名列表（已解析过的文件） | 是 | `["abc123.pdf", "def456.docx"]` |
| message | String | 用户发送的问题内容 | 是 | `"请总结这份文件的核心观点"` |

> 调用方无需传入此前的对话历史消息，后端自动维护对话上下文的连贯性。

### 请求示例

```json
{
  "businessType": "chat",
  "params": {
    "chatId": "conv-20260401-001",
    "fileNames": [
      "abc123def456.pdf",
      "789ghi012jkl.docx"
    ],
    "message": "请总结这两份文件的核心观点"
  }
}
```

### 响应参数说明

| 字段 | 类型 | 说明 |
|------|------|------|
| chatId | String | 对话唯一标识 |
| answer | String | 大模型的回答内容 |
| isNewChat | Boolean | 是否为新创建的对话（首次请求时为 `true`） |

### 响应示例

#### 成功响应（HTTP 200）

```json
{
  "chatId": "conv-20260401-001",
  "answer": "根据文件内容，核心观点包括：1. ...... 2. ......",
  "isNewChat": true
}
```

#### 参数错误（HTTP 400）

```json
{
  "error": "fileNames不能为空"
}
```

#### 引用文件未解析（HTTP 404）

```json
{
  "error": "文件 abc123.pdf 尚未解析，无法用于对话"
}
```

#### 大模型调用失败（HTTP 502）

```json
{
  "error": "大模型服务响应异常"
}
```

## 2. 删除对话接口

**接口路径：** `/llm/chat/delete`  
**请求方法：** `POST`  
**功能说明：** 删除指定对话，同步清理后端资源

### 请求参数说明

| 参数 | 类型 | 说明 | 必填 |
|------|------|------|------|
| businessType | String | 固定值：`"chat"` | 是 |
| params | Object | 删除参数 | 是 |

### params 参数结构

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| chatId | String | 要删除的对话唯一标识 | 是 |

### 请求示例

```json
{
  "businessType": "chat",
  "params": {
    "chatId": "conv-20260401-001"
  }
}
```

### 响应示例

#### 删除成功（HTTP 200）

```json
{
  "chatId": "conv-20260401-001",
  "deleted": true,
  "msg": "对话已删除"
}
```

#### 对话不存在（HTTP 404）

```json
{
  "error": "对话不存在"
}
```

## 3. 调用方职责说明

以下数据与功能由调用方自行维护，本接口不提供相关查询能力。

### 3.1 对话列表管理

调用方应在自身系统中维护对话列表，记录 `chatId`（对话唯一标识）、创建者等信息。

当调用方不再需要某对话时，应调用 `/llm/chat/delete` 通知后端同步释放资源。

### 3.2 对话历史管理

每次调用 `/llm/chat` 时，调用方会获得完整的 `message`（用户问题）与 `answer`（大模型回答），应将其追加存储到本地的对话历史中，用于界面展示和回溯。

> **注意**：后端依赖内部机制维护对话上下文的连贯性，但不提供对话历史查询接口。若调用方未存储历史，将无法在界面上回顾此前的问答内容。
